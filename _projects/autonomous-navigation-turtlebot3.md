## Overview

Autonomous navigation is a fundamental capability for mobile robots operating in structured environments.  
This project investigates a complete navigation pipeline for a **TurtleBot3 mobile robot**, integrating environment mapping, path planning, and trajectory tracking within a simulated indoor scenario.
The environment is first reconstructed through **LIDAR-based SLAM** using the ROS `gmapping` package while manually exploring the environment in the Gazebo simulator.  
The resulting occupancy grid map is then processed in MATLAB to enable path planning using an **optimized implementation of the A\* algorithm**, designed to efficiently handle large grid maps while reducing computational overhead.
Once optimal paths are computed between selected start and goal locations, a **feedback control system based on the Pure Pursuit algorithm** is implemented in Simulink to allow the robot to autonomously follow the planned trajectories.  
The controller communicates with the simulated robot through ROS topics, generating linear and angular velocity commands that ensure stable and accurate trajectory tracking.
The resulting system demonstrates the integration of **mapping, planning, and control modules** within a unified framework for autonomous mobile robot navigation.

## Mapping the Environment

Accurate environmental mapping is a fundamental step in autonomous navigation, as it provides the spatial representation required for path planning and motion control.  
In this project, the environment was mapped using a **TurtleBot3 mobile robot** operating inside the Gazebo simulation environment.  
Mapping was performed through the ROS `gmapping` package, which implements a **SLAM (Simultaneous Localization and Mapping)** algorithm that incrementally builds a 2D occupancy grid map using LIDAR sensor measurements.

### Simulation Environment

The robot operates in the **TurtleBot3 House** environment available in the ROS TurtleBot3 simulation packages.  
The environment represents a small indoor apartment composed of multiple rooms, corridors, and various pieces of furniture acting as obstacles for the robot.

### Mapping Process

The map was generated by manually driving the robot through the environment using the ROS `teleop_node`.  
During exploration, the LIDAR sensor continuously scans the surroundings while the SLAM algorithm estimates the robot pose and incrementally updates the occupancy grid map.
Initial mapping attempts produced maps with limited accuracy, therefore several `gmapping` parameters were adjusted to improve localization stability and mapping quality.

| Parameter | Default value | Modified value |
|----------|---------------|---------------|
| `linearUpdate` | 0.2 | 0.05 |
| `angularUpdate` | 0.5 | 0.01 |
| `minimumScore` | 50 | 200 |
| `particles` | 30 | 200 |

Reducing the update thresholds allowed the system to react more frequently to small robot movements, while increasing the number of particles improved the robustness of the particle filter used by the SLAM algorithm.

### Robot Model

The platform used in this project is the **TurtleBot3**, a compact differential-drive mobile robot widely used for research and education in robotics.  
The robot is equipped with a 2D LIDAR sensor that enables environment perception and SLAM-based mapping.

### Safety Area Definition

To ensure safe navigation during path planning, a **safety area around obstacles** was introduced.  
Since the robot has a non-negligible physical size, planning paths too close to obstacles could result in collisions during execution.
To mitigate this issue, obstacles in the occupancy grid map were **inflated by several pixels**, effectively expanding their boundaries and defining regions that the robot cannot traverse.  
This process creates a safety margin that guarantees feasible and collision-free paths.

## A* Algorithm Implementation

Path planning was performed using the **A\*** algorithm, a widely used search algorithm for computing optimal paths in graph-based environments.  
The algorithm evaluates possible paths between a start node and a goal node by minimizing a cost function that combines the distance already traveled and an estimate of the remaining distance to the goal.
More specifically, A\* evaluates each node according to the function:

<p align="center">

$$
f(n) = g(n) + h(n)
$$

</p>

where:
- **g(n)** represents the cost of the path from the start node to the current node  
- **h(n)** represents a heuristic estimate of the distance from the current node to the goal  
- **f(n)** is the total estimated cost used to guide the search

At each iteration, the algorithm expands the node with the lowest estimated total cost and evaluates its neighboring nodes.  
If a shorter path to a node is found, the cost and predecessor information are updated accordingly.  
This process continues until the goal node is reached, guaranteeing the optimal path provided that the heuristic function is admissible.
In this project, the algorithm operates on a **grid-based representation of the environment**, where each cell of the occupancy map corresponds to a node in the search graph.  
Movement between nodes is allowed in the four cardinal directions and the four diagonal directions, enabling smoother trajectories within the discretized map.

### Map Representation

The occupancy grid map generated through SLAM was imported into MATLAB and converted into a **binary matrix representation**, where:
- free cells correspond to navigable areas
- occupied cells represent obstacles

To simplify coordinate conversions between the map and the Gazebo environment, the map was embedded into a **larger 501×501 grid**, allowing the origin of the simulation coordinate system to coincide with the center of the matrix.
Additionally, obstacles were expanded to include the previously defined **safety margin**, ensuring that the planned paths always maintained a sufficient distance from obstacles.

### Algorithm Optimizations

Because the environment map contains hundreds of thousands of nodes, a straightforward implementation of A\* would lead to high computational costs.  
Several optimizations were therefore introduced to improve both memory efficiency and execution time.

**Removal of the full cost matrix**

Previous implementations relied on a large matrix storing the cost between every pair of nodes.  
Such a matrix would become extremely large for maps of this size.  
Instead, node connectivity is determined dynamically through a function that directly returns the valid neighboring nodes for a given position while ignoring occupied cells.

**Sparse data structures**

Many of the vectors used by the algorithm contain mostly unused elements.  
To reduce memory usage and improve performance, several internal data structures, such as the vectors storing node states, distances, and predecessors, were implemented using **sparse matrices**.  
This approach stores only the elements that are actually used during the search process.

**Sparse map representation**

The occupancy map itself was also represented in sparse form, storing only the obstacle cells.  
This significantly reduces memory consumption when working with large maps and improves the efficiency of obstacle checks during neighbor evaluation.

### Heuristic Function

The heuristic used in the search algorithm is the **diagonal distance** between the current node and the goal.  
This heuristic is consistent with the allowed movement model, which includes both cardinal and diagonal motions.  
By guiding the search toward the goal while preserving admissibility, this heuristic ensures that the algorithm always finds the shortest available path.

### Algorithm Limitations

Despite its effectiveness, the A\* algorithm presents some inherent limitations in the context of robotic navigation:
- **Prior knowledge of the map is required**, meaning the environment must be known before planning begins.
- The algorithm assumes a **static environment**, and therefore cannot directly handle unexpected obstacles or dynamic changes without re-planning.

Nevertheless, when applied to a known environment, the optimized implementation proved capable of efficiently computing optimal paths even for relatively large grid maps.

## Trajectory Tracking

Once the optimal path has been computed through the A* algorithm, the robot must be able to physically follow the planned trajectory.  
To achieve this, a **feedback control system** was implemented in Simulink and integrated with the ROS ecosystem in order to control the robot within the Gazebo simulation.

The controller receives the reference path generated in MATLAB and the real-time robot pose obtained from the `/odom` ROS topic.  
Based on this information, the control system generates the linear and angular velocity commands required to guide the robot along the desired trajectory.

### Feedback Control for Trajectory Tracking

Trajectory tracking is performed using a **Pure Pursuit controller**, a widely used geometric control strategy for mobile robots.  
The controller continuously computes a target point on the reference path located at a fixed **look-ahead distance** from the current robot position.  
Using this target point, the algorithm determines the linear and angular velocities necessary to steer the robot toward the path.

The controller outputs the velocity commands which are published to the `/cmd_vel` ROS topic, allowing the simulated robot to execute the motion in Gazebo.

The following control parameters were selected after experimental tuning to ensure stable and smooth motion:
- **Desired linear velocity:** 0.12 m/s  
- **Maximum angular velocity:** 0.35 rad/s  
- **Lookahead distance:** 0.25 m

To further improve motion stability, two additional mechanisms were introduced:
- **Smooth acceleration:** during the first seconds of motion the velocity gradually increases from zero to the desired value. This prevents abrupt starts and reduces trajectory deviation when the robot initially reorients itself.
- **Smooth deceleration:** as the robot approaches the goal, the velocity is progressively reduced based on the remaining distance. A tolerance radius of **3 cm** is used to stop the robot once the target region is reached.

These mechanisms ensure that the robot follows the trajectory smoothly while maintaining stable motion throughout the navigation task.

## Discussion and Conclusions

This project demonstrates a complete navigation pipeline for a mobile robot in a simulated indoor environment, combining **SLAM-based mapping, A* path planning, and trajectory tracking control**.

The environment was first mapped using the ROS `gmapping` package, generating an occupancy grid map used for path planning.  
An optimized implementation of the **A* algorithm** was then used to compute collision-free paths across the environment while maintaining reasonable computational performance even on large maps.

To enable autonomous motion, a **Pure Pursuit feedback controller** was implemented in Simulink and integrated with ROS to track the planned trajectories in Gazebo.  
Simulation results show that the robot can follow the reference paths with good accuracy and stable motion.

Although the approach works effectively in known environments, the reliance on a precomputed map and static obstacles represents a limitation.  
Future improvements could focus on **dynamic obstacle avoidance and real-time path replanning** to handle more complex and changing environments.
